# Обмен рублями — Telegram Mini App + бот + подписка

MVP-приложение для площадки обмена рублями внутри Telegram:

- Авторизация пользователей через их аккаунт Telegram (без паролей, автоматически, по данным Mini App).
- Вкладки по суммам: **до 50к / 50–100к / 100–200к / 200к+**.
- Объявления «хочу купить рубли» / «хочу продать рубли».
- Платная подписка (по умолчанию 990 ₽ / 30 дней) через **ЮKassa** — без активной подписки нельзя ни разместить объявление, ни увидеть контакт автора.
- Telegram-бот, который открывает Mini App и показывает статус подписки.

Это рабочий MVP, который нужно задеплоить на свой сервер и донастроить под себя (см. "Что дальше" в конце).

## Из чего состоит проект

```
rub-p2p/
├── backend/        # FastAPI: API объявлений, оплаты, проверка Telegram-авторизации
│   ├── app/
│   └── tests/       # unit- и интеграционные тесты (pytest)
├── bot/             # Telegram-бот (aiogram): кнопка "Открыть биржу", /status, /admin_grant
├── webapp/          # Mini App (чистые HTML/CSS/JS + Telegram WebApp SDK)
├── docker-compose.yml
└── .env.example
```

Backend и бот используют общую SQLite-базу (файл `data/app.db`, том Docker). Для продакшена с заметной нагрузкой замените `DATABASE_URL` на Postgres — код на SQLAlchemy, менять почти ничего не придётся.

## Что нужно подготовить заранее

1. **Токен бота.** В Telegram напишите **@BotFather** → `/newbot` → следуйте инструкциям → получите токен вида `123456789:AA...`.
2. **Домен с HTTPS.** Telegram Mini App обязательно открывается по `https://`. Нужен любой домен, привязанный к вашему серверу (Cloudflare/Reg.ru/Timeweb и т.п.) — self-signed сертификаты не подойдут.
3. **Сервер (VPS).** Подойдёт любой VPS с Ubuntu 22.04+ и установленным Docker + Docker Compose (2 ГБ RAM достаточно для старта).
4. **Магазин в ЮKassa.** Зарегистрируйтесь на https://yookassa.ru — для приёма оплаты картами понадобится оформленный магазин (ИП, ООО или самозанятый — у ЮKassa есть отдельные условия для каждого случая). Это финансовый/юридический вопрос — я не даю юридических консультаций, уточните требования и вопросы фискализации (54-ФЗ, чеки) напрямую в поддержке ЮKassa или у бухгалтера/юриста. После регистрации в личном кабинете ЮKassa возьмите **shopId** и **secretKey** (Настройки → API ключи).

## Установка на сервере

```bash
git clone <ваш-репозиторий-или-скопируйте-папку-rub-p2p> && cd rub-p2p
cp .env.example .env
nano .env   # заполните BOT_TOKEN, BOT_USERNAME, ADMIN_IDS, WEBAPP_URL, YOOKASSA_*
```

Поднимите HTTPS-реверс-прокси перед контейнером backend (проще всего — **Caddy**, он сам выпускает сертификат Let's Encrypt):

```
# /etc/caddy/Caddyfile
your-domain.example.com {
    reverse_proxy localhost:8000
}
```

```bash
sudo apt install -y caddy
sudo systemctl restart caddy
```

Запустите приложение:

```bash
docker compose up -d --build
docker compose logs -f
```

Backend будет слушать `127.0.0.1:8000` (проксируется Caddy на ваш домен), бот подключится к Telegram сам (long polling, входящий порт не нужен).

## Настройка бота в BotFather

1. `/mybots` → выберите бота → **Bot Settings → Menu Button** → укажите `https://your-domain.example.com` — тогда кнопка Mini App появится в меню чата с ботом.
2. Если хотите, чтобы бот отвечал в группах трейдеров: `/setjoingroups` → Enable, и добавьте бота в нужные чаты — он не читает историю чатов, а просто доступен по кнопке/командам.
3. Проверьте: напишите боту `/start` — должна появиться кнопка "Открыть биржу", открывающая Mini App.

## Настройка вебхука ЮKassa

В личном кабинете ЮKassa: **Настройки → HTTP-уведомления** → укажите URL:

```
https://your-domain.example.com/api/yookassa/webhook
```

Это нужно, чтобы подписка активировалась автоматически сразу после оплаты, а не только когда пользователь вернётся в приложение (хотя мы дополнительно подстраховались поллингом статуса на фронтенде — см. `webapp/app.js`).

## Ручная выдача подписки (без оплаты)

Полезно для тестов или для VIP-пользователей. В `.env` укажите свой Telegram ID в `ADMIN_IDS` (узнать свой ID можно у **@userinfobot**), затем в чате с ботом:

```
/admin_grant 123456789 30
```

выдаст пользователю с ID `123456789` подписку на 30 дней.

## Локальный запуск для разработки (без Docker)

```bash
cd backend
python3 -m venv venv && source venv/bin/activate
pip install -r requirements.txt
cp ../.env.example ../.env   # и заполните хотя бы BOT_TOKEN
uvicorn app.main:app --reload --port 8000
```

Открыть Mini App локально напрямую из браузера не получится с полноценной Telegram-авторизацией (initData подписывается только настоящим Telegram-клиентом) — для локальной отладки UI можно временно замокать `tg.initData` в `webapp/app.js` или тестировать через **ngrok**:

```bash
ngrok http 8000
# полученный https-адрес указать как WEBAPP_URL и Menu Button в BotFather
```

## Тесты

```bash
cd backend
pip install -r requirements.txt pytest
pytest -v
```

Тесты покрывают: проверку подписи Telegram initData (валидная/подделанная/просроченная/из чужого бота), попадание суммы в нужную вкладку, требование подписки для публикации объявлений, скрытие контактов для неподписанных пользователей, сценарий оплаты через мок ЮKassa (создание платежа → вебхук → активация подписки).

> В текущей песочнице, где собирался этот проект, нет доступа к PyPI, поэтому полный прогон через `pytest` я не смог выполнить здесь — но модуль проверки подписи Telegram (`app/security.py`), самый критичный для безопасности, я протестировал напрямую через stdlib (без внешних пакетов) и он проходит все проверки. Прогоните `pytest -v` на своей машине или в CI перед продакшн-деплоем — это стандартный шаг разработки, а не формальность.

## Модель безопасности (коротко)

- Никто не может представиться другим пользователем: `initData` подписан HMAC-SHA256 секретом, который знает только Telegram и ваш бот (`BOT_TOKEN`). Бэкенд пересчитывает подпись и сверяет её при каждом запросе (`app/security.py`, `app/deps.py`).
- Статус оплаты подписки бэкенд не берёт из тела вебхука "на слово" — он дополнительно запрашивает актуальный статус платежа у самой ЮKassa через API, прежде чем активировать подписку. Это защищает от поддельных POST-запросов на адрес вебхука.

## Что стоит доработать дальше (не входит в этот MVP)

- **Модерация объявлений и антиспам** — сейчас любой подписчик может опубликовать что угодно; для чатов с реальными деньгами обычно нужна ручная/автоматическая модерация и жалобы на пользователей.
- **Отзывы/рейтинг трейдеров** — чтобы отличать проверенных людей от новых аккаунтов (частая функция таких площадок).
- **Переход на Postgres** и вынесение медиа/логов, если нагрузка вырастет — SQLite в MVP достаточно для старта.
- **Несколько тарифов подписки** (например, недельная/месячная/годовая) — сейчас один тариф, задан в `.env`.
- **Учёт валютной пары/курса** — сейчас курс — это просто свободный текст в поле "детали"; если нужно строгое поле с курсом крипто/рубль, это легко добавить отдельным полем.
